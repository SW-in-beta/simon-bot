# simon-oncall

온콜 문의 분석 스킬. 슬랙으로 들어온 기술 문의를 파악하고, 관련 코드와 사내 자료를
**심층 분석(simon-study) → 원인/해결 방향 → 슬랙 답변 초안**까지 제공합니다.

## Why this exists

온콜 문의는 대부분 "이거 왜 이런 거야?", "이게 현재 가능할까요?" 형태입니다.
좋은 답변을 하려면 슬랙 맥락 파악 → 최신 코드 분석 → 사내 문서/논의 검색 →
종합 판단이 필요한데, 이 과정을 체계적으로 수행합니다.

## Input

사용자가 제공하는 것:
1. **슬랙 URL** — 온콜 문의 메시지 (buzzvil.slack.com)
2. **레포 이름** (선택) — 관련 서비스의 레포지토리 (~/buzzvil/{repo_name}에 위치)

슬랙 URL이 빠져 있으면 요청합니다.
레포 이름이 빠져 있으면 `references/repo-map.md`를 참조하여 문의 키워드로 관련 레포를 추천합니다.
추천이 명확하지 않을 때만 사용자에게 확인합니다.

## Instructions

### Phase 1: 문의 파악

슬랙 스레드를 읽고 문의의 핵심을 파악합니다.

```bash
~/.claude/skills/buzzvil-slack/scripts/fetch_thread.sh "{slack_url}" --resolve-users
```

단일 메시지(스레드 없음)라면:
```bash
~/.claude/skills/buzzvil-slack/scripts/fetch_message.sh "{slack_url}"
```

스레드에서 추출할 것:
- **문의자**: 누가 물어봤는가
- **핵심 질문**: 한 문장으로 요약
- **문의 유형**:
  - `ERROR` — 에러/장애 ("왜 500이 나와요?", "이거 깨졌어요")
  - `DATA_ANOMALY` — 데이터 이상 ("수치가 이상해요", "왜 0이 나오지?")
  - `LOGIC_INQUIRY` — 로직/동작 질문 ("이 로직이 왜 이렇게 동작해요?")
  - `CAPABILITY` — 기능 가능 여부 ("이런 게 가능할까요?")
- **추가 맥락**: 이미 논의된 내용, 첨부 스크린샷/로그, 언급된 키워드

사용자에게 파악한 내용을 간단히 확인합니다:
> "문의 내용을 이렇게 이해했습니다: [요약]. 맞나요?"

### Phase 1.5: 레포 특정

사용자가 레포를 명시하지 않았으면 `references/repo-map.md`를 읽고, Phase 1에서 파악한 키워드와 대조하여 관련 레포를 특정합니다.

- 매핑이 명확하면 (1개 레포로 좁혀지면) 바로 진행하고 사용자에게 알림: "관련 레포로 {repo_name}을 선택했습니다"
- 후보가 2-3개면 사용자에게 선택지 제시: "A, B, C 중 어디를 먼저 볼까요?"
- 로컬에 레포가 없으면 클론: `gh repo clone Buzzvil/{repo_name} ~/buzzvil/{repo_name} -- --depth 1`

### Phase 2: 레포 준비

최신 master(또는 main) 브랜치 기준으로 분석해야 합니다.

```bash
cd ~/buzzvil/{repo_name}
git fetch origin
# master 우선, 없으면 main
git checkout master 2>/dev/null || git checkout main
git pull origin $(git branch --show-current)
```

레포의 CLAUDE.md 또는 AGENTS.md가 있으면 읽어서 프로젝트 구조와 컨벤션을 파악합니다.

### Phase 3: 심층 분석

이 단계가 핵심입니다. 세 개의 서브페이즈로 진행합니다:

- **3-A** 개인 지식 베이스 사전 조회
- **3-B** simon-study 서브에이전트 심층 분석
- **3-C** *(조건부)* Flow Verification Expert로 가설 교차 검증

#### Phase 3-A: 개인 지식 베이스 사전 조회

Obsidian wiki vault(`~/Obsidian/*-wiki/`)에서 문의 키워드, 레포명, 에러 유형으로 Grep 검색하여 과거 온콜 분석이나 관련 지식이 있는지 확인합니다. 발견된 기존 지식은 3-B의 simon-study 서브에이전트에 추가 맥락으로 전달합니다. 없으면 skip.

#### Phase 3-B: simon-study 심층 분석

**simon-study를 서브에이전트로 실행**하여 깊이 있는 분석을 수행합니다.

Agent 도구로 서브에이전트를 생성하고, 아래 내용을 프롬프트로 전달합니다:

```
아래 문제를 simon-study 스킬(/simon-study)을 사용하여 분석해줘.

## 문제
{Phase 1에서 파악한 핵심 질문}

## 맥락
{슬랙 스레드에서 파악한 추가 맥락 — 문의자가 언급한 구체적 현상, 에러 메시지, 데이터 값 등}

## 레포
~/buzzvil/{repo_name} (현재 최신 master/main 체크아웃 완료)

## 분석 요청사항
- graphify-out/GRAPH_REPORT.md가 존재하면 먼저 읽고, god nodes와 communities를 탐색 컨텍스트로 활용한다. `graphify query`로 에러 관련 모듈을 탐색하고 `graphify path`로 장애 전파 경로를 추적한다 (상세: `~/.claude/skills/_shared/graphify-context.md`)
- 관련 코드 경로와 동작 원리를 file:line 형식으로 구체적으로
- **사용 흐름 전체 검증 (필수 원칙)**: 함수/심볼의 동작을 분석할 때 바디 한 군데만 보고 단정하지 않는다. 해당 심볼이 **실제로 어떻게 생성·주입·호출·사용되는지 사용 흐름 전체**를 검증한다. DI 체인(생성자 파라미터·wiring)이 끊긴 채 silent fallback으로 wrong value가 표출되는 버그는 함수 바디만 봐서는 잡히지 않는다.
  1. **구현체 위치 확인**: 인터페이스 → 실제 struct 구현체 grep
  2. **생성자 시그니처 직접 Read**: consumer의 `New` 함수 파라미터 목록에 해당 의존성이 포함되어 있는지 확인 (이름 유사성으로 추정 금지, 실제 Read 필수)
  3. **caller/wrapper 반환값 추적**: 반환값이 타입 캐스팅·래핑·변환되며 최종 사용처까지 어떻게 전달되는지 grep
  4. **silent fallback 확인**: `if p == nil { return default }` 패턴 — DI 누락이 런타임 에러 없이 0/nil/default로 표출되는 지점
  5. **전수 호출부 검사**: consumer `New` / factory 호출부 전체를 grep으로 찾아 모든 위치에서 해당 의존성이 전달되는지 확인 (main.go, wire.go, factory.go 등 wiring 파일 포함)

  트리거: Provider/Factory/DI/Dependency/wiring/주입 키워드 등장, 또는 특정 값이 항상 0·nil·default로 관찰되는 현상.
- **복수 경로/가설 탐색 (필수 원칙)**: 가설은 절대 1개로 닫지 않는다. 코드 분석 결과가 그럴듯해 보여도 "이게 유일한 설명인가?"를 명시적으로 자문하고, 다음을 의도적으로 수행한다.
  1. **대안 가설 최소 1개 탐색**: 동일 현상을 설명할 수 있는 다른 메커니즘을 최소 1개 더 떠올리고, 코드/문서에서 지지·반박 근거를 모두 수집
  2. **제2 처리 경로 grep 의무**: "동일한 데이터/요청을 처리하는 다른 코드는 없는가?"를 명시적으로 검색 — 같은 도메인의 다른 모듈(예: tracker 종류별), middleware/후처리/wrapper 체인, third-party 통합 디렉토리(`thirdparty/`, `integrations/`, `vendors/` 등), 동일 키워드의 별도 디렉토리 grep 재실행
  3. **가설 도출 직후 실측값 재확인**: 슬랙 스레드의 첨부 URL·로그·스크린샷에 가설과 대조 가능한 실측값이 있는지 다시 본다. 있다면 가설이 예측한 값과 일치하는지 명시적으로 비교
  4. **단일 가설 종결 시 태그 의무**: 끝까지 1개 가설만 남았다면 보고서에 `[단일 가설 — 대안 미탐색]` 태그를 부착하고, 해결 방향에 "추가 탐색 필요" 항목을 포함

  트리거: 모든 분석에 적용 (skip 조건 없음). "그럴듯한 가설 1개" 자체가 confirmation bias의 신호다.
- Confluence에서 관련 RFC/설계 문서 검색
- Slack에서 관련 이전 논의 검색
- 설계 트레이드오프 (왜 이렇게 만들었는지)
- 해결/개선 방향 2-3가지 (각각의 복잡도와 영향 범위 포함)
- 문제가 데이터와 관련된 경우(DATA_ANOMALY, 수치 이상, 데이터 확인 필요), /buzzvil-data 스킬로 실제 데이터를 조회하여 현상을 검증. **반드시 prod 환경 테이블(prod_* prefix)을 사용** — 온콜 문의는 운영 환경 기준이므로 dev_ 테이블이 아닌 prod_ 테이블을 조회해야 한다. 조회한 테이블, SQL 쿼리, 기간, 건수를 투명하게 기록
```

서브에이전트가 simon-study를 통해 수행하는 것:
- 질문 체인 구성 → 개념/코드 병렬 리서치 → Confluence·Slack 검색 → 데이터 조회(해당 시) → 보고서 작성

#### Phase 3-C: Flow Verification (조건부)

simon-study의 root cause 가설을 독립 전문가 Agent가 **반박 시도** 관점에서 교차 검증합니다. 같은 모델이 바디 단일 지점만 보고 내린 판단을 뒤집는 것이 목적입니다.

**트리거 조건** (하나라도 해당되면 반드시 spawn):
- 문의 또는 simon-study 가설에 Provider / Factory / DI / Dependency / wiring / 주입 / 의존성 키워드 등장
- 현상이 "항상 0·nil·default·빈 값"으로 표출 (silent fallback 의심)
- simon-study root cause가 인프라(cache·sync·DB) 쪽으로 지목된 경우 (코드 쪽 가능성을 교차 확인)

트리거에 해당하지 않으면 3-C는 skip하고 Phase 4로 진행합니다.

**프로토콜**: `references/flow-verification-expert.md` 참조.

**Verdict 처리 요약**:
- `CONFIRMED` → simon-study 가설대로 Phase 4 진행. 보고서에 `Flow Verification: CONFIRMED` 명시
- `CONTRADICTED` → `corrected_root_cause`를 새 root cause로 채택. 보고서 "원인 분석" 섹션에 초기 가설·뒤집힌 단계·수정된 root cause·누락된 injection points를 모두 기록
- `PARTIAL` → 수정된 root cause로 Phase 4 진행, 양쪽 가설과 근거 명시

Verdict 원문은 `~/claude-reports/oncall-{ts}/flow-verdict.md`에 저장하여 감사 추적을 남깁니다.

Phase 3 완료 후, 결과를 기반으로 Phase 4로 진행합니다.

### Phase 4: 산출물 작성

두 가지 산출물을 작성합니다.

#### 4-1. 심층 분석 보고서

simon-study 결과를 온콜 맥락에 맞게 재구성합니다:

```markdown
# 온콜 문의 분석: {핵심 질문 한 줄 요약}

| 항목 | 내용 |
|------|------|
| 문의자 | {이름} |
| 유형 | {ERROR / DATA_ANOMALY / LOGIC_INQUIRY / CAPABILITY} |
| 분석일 | {날짜} |
| 레포 | {repo_name} |
| 원문 | {slack_url} |

## 핵심 멘탈 모델
{이 문제를 이해하는 데 필요한 사고 프레임워크 — simon-study에서 추출}

## 코드 분석
{관련 코드 경로, 데이터 흐름 — mermaid 다이어그램 포함}

## 설계 트레이드오프
{현재 구현의 의도적 선택과 그 한계}

## 데이터 분석 (데이터 조회 수행 시)
{조회한 데이터의 핵심 결과 — 수치, 분포, 추이}

### 데이터 조회 내역
| 목적 | 테이블 | 쿼리 요약 | 기간 | 건수 | 결과 | 신뢰도 참고 |
|------|--------|-----------|------|------|------|-------------|
| {무엇을 확인} | {테이블명} | {SQL 핵심부} | {날짜 범위} | {N건} | {핵심 수치} | {데이터 지연/제약사항} |

## 원인 분석 (또는 답변 근거)

**확신도 태그 필수** — 다음 중 정확히 1개를 원인 분석 첫 줄에 부착:

- `[검증됨 — 코드+데이터]`: 코드 경로 + 실측 데이터(서빙 URL/로그/메트릭/재현) 양쪽으로 확인됨. 단정 표현 허용.
- `[검증됨 — 코드만]`: 복수 경로/대안 가설 탐색 후 코드만으로 확인됨 (실측 데이터 미확보). 보고서·슬랙 답변에서 hedge 표현 사용.
- `[추정 — 가설]`: 코드 분석 기반 가설, 실측 미확인. "추정"임을 본문에 명시.
- `[단일 가설 — 대안 미탐색]`: 대안 가설/제2 처리 경로 탐색이 미흡. 해결 방향에 "추가 탐색 필요" 항목을 반드시 포함.

{ERROR/DATA_ANOMALY: 근본 원인 — 데이터 증거 포함}
{LOGIC_INQUIRY: 왜 이렇게 동작하는지}
{CAPABILITY: 가능/불가능 여부와 근거}

## 해결 방향
{2-3가지 방안, 각각의 장단점·복잡도·영향 범위}

## 참고 자료
{Confluence 링크, Slack 스레드 링크, 외부 문서 — 출처 URL 필수}
```

#### 4-2. 요약 + 슬랙 답변 초안

```markdown
## 요약 (3줄)
1. {문제가 뭔지 / 질문의 핵심}
2. {왜 발생하는지 / 왜 그렇게 동작하는지 / 가능 여부}
3. {해결·대응 방향}

## 슬랙 답변 초안
{문의자에게 보낼 답변}
```

**슬랙 답변 작성 원칙:**
- 결론을 먼저, 근거를 나중에
- 문의 유형에 맞는 톤: ERROR는 상황 인지 + 원인 + 조치 방향, CAPABILITY는 가능/불가능을 명확히
- 코드 위치(`file:line`)나 설정값을 구체적으로 언급하면 신뢰도가 올라감
- 불확실한 부분은 "추가 확인이 필요합니다"로 솔직하게
- 간결하게 — 스레드에서 읽기 좋은 길이로. 상세 내용이 필요하면 "자세한 내용은 별도로 공유드리겠습니다"
- **확신도 톤다운 (기본 모드)**: 단정 표현("~로 확인됩니다", "~가 원인입니다", "~하시면 됩니다")은 원인 분석 태그가 `[검증됨 — 코드+데이터]`인 경우에만 사용한다. 그 외 모든 경우는 hedge 표현 사용:
  - 원인: "현 분석으로는 ~로 보입니다", "코드 상으로는 ~로 추정됩니다"
  - 액션: "단정 전에 실제 ~ 확인이 필요해 보입니다", "~ 가능성이 있어 보이니 ~ 확인 후 진행 부탁드립니다"
- **외부 전달 액션 가드**: 광고주 요청·외부 설정 변경·배포·되돌리기 어려운 액션 아이템이 가설 기반(`[추정 — 가설]` 또는 `[단일 가설 — 대안 미탐색]`)이면, "현 가설로는 X가 필요해 보이나, 실제 동작/실측값을 함께 확인 후 진행하시는 것이 안전합니다" 형태의 게이트 문구를 답변에 포함시킨다. 가설을 사실처럼 단정한 액션을 외부에 그대로 보내지 않는다.

### Phase 4-3: 보고서 저장

심층 분석 보고서를 `~/claude-reports/`에 저장한다.

```bash
mkdir -p ~/claude-reports
REPORT_FILE="$HOME/claude-reports/oncall-$(date +%Y%m%d-%H%M%S).md"
# 보고서 마크다운을 REPORT_FILE 경로에 저장
```

저장 완료 후 사용자에게 안내한다:
> "보고서가 `{REPORT_FILE}`에 저장되었습니다.
> - Obsidian에서 리뷰하려면 `/simon-md-reviewer`를 호출하세요.
> - 개인 지식 베이스(wiki)에 저장하려면 `/simon-brain-update`를 호출하세요."

이후 터미널에 요약 + 슬랙 답변 초안을 출력한다.

### Phase 5: 추가 분석 (사용자 요청 시)

사용자의 후속 요청에 대응합니다:
- "이 부분 더 깊이 봐줘" → 해당 영역 추가 분석
- "답변 톤 수정해줘" → 슬랙 답변 조정
- "실제로 코드 고칠 수 있어?" → simon이나 investigate 스킬 연계 제안
- "모니터링 데이터도 봐야 할 것 같아" → 사용자에게 Datadog/Grafana 대시보드 URL이나 스크린샷 요청

## 주의사항

- **반드시 최신 master/main 기준**: 분석 전 `git fetch && git pull`로 최신 코드를 확보
- **출처 명시**: Confluence 페이지, Slack 스레드를 인용했으면 반드시 URL 포함
- **불확실함은 솔직하게**: 코드에서 확인할 수 없는 추론은 "추정"임을 명시
- **단일 가설 종결 금지**: 첫 가설이 그럴듯해 보여도 "이게 유일한 설명인가?", "동일 데이터를 처리하는 다른 코드 경로(다른 모듈/middleware/third-party 통합)는 없는가?"를 한 번 더 묻고 의도적으로 grep. 단일 가설로 보고서 진입 시 `[단일 가설 — 대안 미탐색]` 태그 부착 의무 (해결 방향에 "추가 탐색 필요" 항목 포함)
- **확신도 보존 (단정 자제)**: 단정 표현은 코드+실측 데이터 양쪽이 검증된 경우(`[검증됨 — 코드+데이터]`)에만 사용. 그 외에는 hedge 표현으로 톤다운. 보고서·슬랙 답변·요약 모두 동일하게 적용
- **과도한 분석 지양**: 문의가 단순하면 보고서도 간결하게. 분량이 아니라 정확도가 중요
